서버 사이드 애플리케이션이란 애플리케이션의 페이지를 서버에서 `렌더링`(HTML, CSS, 이미지, 멀티미디어 자원, 자바스크립트 등을 결합)해서 클라이언트에 전송하는 것이다. 반대로 클라이언트 사이드 애플리케이션은 사용자 인터페이스를 클라이언트에서 렌더링하는 것이다. 즉, 일단 브라우저가 초기(일반적으로 아주 최소한의) HTML을 받으면, 자바스크립트를 사용해 DOM을 동적으로 수정하므로 새 페이지를 표시할 때 서버에 의존하지 않는다(물론 데이터 자체는 서버에서 받는다).

> 💡 **렌더링**(rendering)  
> HTML, CSS, 자바스크립트로 작성된 문서를 파싱(해석)하여 브라우저에 시각적으로 출력하는 것

1999년 이전에는 서버 사이드 애플리케이션이 표준이었다. 웹 애플리케이션이라는 용어 자체는 공식적으로 1999년에 등장했다. 2012년에 접어들며 **스마트폰**이 실생활에 뿌리를 내리면서, **네트워크를 통해 전송하는 정보를 최소한**으로 줄이는 것이 일반화되었고 이에 따라 클라이언트 사이드 애플리케이션이 발달했다.

서버 사이드 애플리케이션은 **서버 사이드 렌더링**(SSR)이라 부를 때가 많고, 클라이언트 사이드 애플리케이션은 **단일 페이지 애플리케이션**(SPA)이라 부를 때가 많다. 클라이언트 사이드 애플리케이션은 리액트, 앵귤러, 뷰 같은 프레임워크를 사용하곤 한다. 결국 두 애플리케이션 간의 차이는 **페이지 전체를 서버에서 보내느냐, 클라리언트에서 동적으로 렌더링 하느냐**에 있다.

현실에서는 서버 사이드와 클라이언트 사이드 애플리케이션을 명백히 구분하기가 어려울 때가 많다. 클라이언트 사이드 애플리케이션은 보통 두세 개의 HTML을 클라이언트로 전송한다. 예를 들어 공개되는 인터페이스와 로그인된 사용자를 위한 인터페이스, 관리자 인터페이스 등이다. 또한 SPA라고 해도 `검색엔진 최적화(SEO)`와 첫 번째 페이지의 성능을 위해 SSR과 결합해 사용하는 경우도 많다.

> 💡 **검색엔진 최적화**(search engine optimization)  
> 웹 페이지 검색엔진이 자료를 수집하고 순위를 매기는 방식에 맞게 페이지를 구성하여 검색 결과의 상위에 나올 수 있도록 하는 작업이다. 방문 트래픽을 증가시켜 인터넷 마케팅의 수단으로도 활용된다.

> 🤔 SPA(CSR) 방식에서 SEO가 어려운 이유  
> 웹에서 컨텐츠(데이터)를 크롤링하는 검색엔진의 봇들은 단순히 **HTML 파일만 렌더링**하고 자바스크립트 파일은 실행하지 않는다. 따라서 HTML 파일은 최소한으로 보내는 CSR 방식에서는 검색엔진 봇들이 컨텐츠를 제대로 수집하지 못한다.

일반적으로 서버에서 HTML 파일을 1~3개 정도 보내고 사용자는 동적인 DOM 조작을 통해 생성되는 인터페이스를 사용한다면 이를 클라이언트 사이드 렌더링이라고 말할 수 있다. 다양한 뷰에서 사용하는 데이터(보통 JSON 형태)와 멀티미디어 자원은 여전히 네트워크를 통해 전송되어야 한다.
